## 2. Critical review — risks & issues (verified in code)

### High severity

- **H1 — Batch codec (default!) races on the shared stream.** The background jthread reads
  `stream_` (`file_codec_packet_lz4_batch001.h:409-489`) while the main thread polls
  `stream_.good()` every row (`reader.hpp:210`). Data race (UB) plus two functional failures:
  a transient failbit window can end `readNext()` mid-file nondeterministically, and on
  footer-less (crashed) files the BG EOF path leaves eofbit set, **silently dropping the last
  complete packet** — defeating the crash-recovery guarantee. Fix: route stream state through the
  codec; BG thread restores stream state, EOF only via `bg_has_next_packet_`.
- **H2 — Data race on `bg_exception_`.** `rethrowBgException()` is called from `readRow()` /
  `beginWrite()` without waiting for BG idle while the BG thread may assign it. Unsynchronized
  `std::exception_ptr` read/write is UB and can miss errors. Move under the mutex / after
  `waitForBgIdle()`.
- **H3 — `MAX_ROW_LENGTH` enforced on read only.** No write path checks the serialized row length
  (all 4 read paths throw at 16 MiB−2). Wide string layouts can write files the reader rejects;
  a row length colliding with `PCKT_TERMINATOR` (0x3FFFFFFF) misparses as packet terminator.
  `writeRow()` should throw.

### Medium severity

- **M1 — Delta002 decode shift UB on hostile input**: 4-bit header codes 10–15 yield
  `deltaBytes` up to 14 → `<< (i*8)` up to 104 bits (`row_codec_delta002.hpp:51-56, 454-461`).
  Bounds-checked, so no OOB, but UB + column desync. Add `deltaBytes > sizeof(T)` → throw.
- **M2 — Direct-access reads skip packet checksum validation** (`reader.hpp:371-418`,
  `file_codec_packet001.h:199-216`): random access returns rows from packets sequential reads
  would reject.
- **M3 — Flat001 writes uninitialized heap bytes for strings > 65 535 B**: buffer sized with full
  `str.size()`, only clamped length written, span not trimmed (`row_codec_flat001.hpp:79-121`) —
  info-leak to disk (ZoH/Delta trim correctly). Reachable via `row.ref<std::string>()`.
- **M4 — Allocation bombs**: batch header sizes only checked vs 1 GiB each (~2 GiB alloc from a
  40-byte file); `FileFooter` `indexSize = start_offset − 28` underflows in `size_t` for
  `start_offset < 28` (`file_footer.h:261-270`).
- **M5 — Format is native-endian, docs say little-endian.** Magic check rejects cross-endian
  files loudly (no misparse), but the portability claim only holds on LE hosts. Add
  `static_assert(std::endian::native == std::endian::little)` or real conversion.
- **M6 — Batch writer can swallow a BG write failure** when the last packet fails and no rows
  follow: `finalize()` skips the rethrow branch, writes a clean footer over a corrupt tail.
  Call `rethrowBgException()` unconditionally after the final wait.

Solid (verified): row-codec deserialization bounds checks are systematic; LZ4 uses only `_safe`
variants; header limits enforced; no memory-corruption primitive found from hostile files.
Note `docs/THREAD_SAFETY.md` declares the library single-threaded — H1/H2 are the library's *own*
internal thread, so they're squarely in scope.

### NaN / ±Inf status (asked as a development item)

The binary core is already almost fully NaN/Inf-safe — bit-exact round-trips through flat001,
zoh001-dynamic (memcmp), delta002-dynamic (memcmp + XOR path). Remaining gaps:

| Gap | Where | Effect |
|---|---|---|
| Static-layout ZoH/Delta use `operator==`/`!=` for change detection | `row_codec_zoh001.hpp:444-453`, `row_codec_delta002.hpp:697` | **−0.0 silently becomes +0.0** (real bug); NaN never "holds" (compression loss only) |
| csv2bcsv float-compat check | `csv2bcsv.cpp:256-260` | one NaN forces column to DOUBLE (minor) |
| **Python `write_dataframe` coerces NaN → 0.0** | `pandas_utils.py:134-167` | the main blocker: pandas path corrupts NaN (with warning) |
| pybcsv parquet import rejects Arrow nulls | `parquet_utils.py:670-691` | pandas→parquet stores NaN as null → rejected |
| Test coverage | — | no NaN/Inf tests for zoh001/flat001/static-delta; no −0.0 tests anywhere |

"Enable NaN and Inf" is therefore cheap: bit-compare (`std::bit_cast`) in the two static-layout
sites, `std::isfinite` guard in csv2bcsv, a `nan_policy` for `write_dataframe` (preserve by
default for float columns), plus a test matrix (NaN payloads, ±Inf, −0.0, subnormals × 3 codecs ×
2 layouts). CSV bridge already round-trips `nan`/`inf` (tested). bcsvCast handles non-finite
deliberately and correctly.

## 3. BCSV vs Parquet head-to-head (same datasets)

Method: identical data (deterministic profile generators) run through both stacks, single-threaded,
pinned, 3 iterations, medians; 500 k rows; pure I/O timing (no validation in loops).
BCSV: flat/zoh/delta row codecs × batch-LZ4 (default, level 1, 64 KB packets), read = readNext +
visitConst over accessed cells. Parquet: pyarrow 24 (Arrow C++), snappy/zstd/lz4, default row
groups; read = `pq.read_table` (+ column projection / `take` / filter). Krow/s = total dataset
rows / elapsed. Harness + raw data: `tmp/h2h/` (report.md, report_ts.md).
Caveat: pyarrow parquet also scales with threads (`use_threads=True`); numbers here are the
apples-to-apples single-thread comparison — BCSV has no multithreaded read path to compare.

### Where each side leads

**BCSV leads** (single-threaded):

- **Full-scan reads**: 1.2–3.5× faster on 14/15 profiles (both datasets); extreme on
  high-cardinality strings (2.6–8.8 M vs 0.4–1.6 M rows/s — Arrow string materialization is
  expensive).
- **Writes on time-series data**: 1.5–3× faster on 13/15 profiles (delta+FoC encodes
  constant-rate/smooth channels nearly for free); plus streaming row-wise writes with bounded
  latency, which parquet fundamentally does not offer (row groups must buffer + column-shred).
- **Numeric/smooth compression**: delta beats best-parquet on smooth numeric data
  (simulation_smooth 20.8 % vs 37.2 %, arithmetic_wide 13.0 % vs 39.5 %, mixed_generic 6.8 % vs
  25.0 %, rtl_waveform 1.0 % vs 0.4 %≈tie); idle/ZoH data compresses to ~0.
- **Sparse-row scans and untuned predicate scans**: with default parquet row groups (no
  pushdown possible), bcsv sequential skip is 2–3× faster.
- Qualitative: crash recovery to last packet, O(log n) row seeks via footer index, embedded
  footprint, no schema files.

**Parquet leads:**

1. **Sparse-column reads: 10–50× (the structural gap).** Columnar chunks mean reading 1 of 84
   columns touches ~1/84 of the file. BCSV must decode whole rows (delta/ZoH state chains make
   column skipping impossible in the current wire format); the gap persists on every profile and
   both datasets.
2. **String-heavy compression: ~2–3× smaller.** Dictionary encoding + zstd: event_log 15.3 % vs
   43.9 %, financial_orders 7.5–17 % vs 21–36 %, string_heavy 25 % vs 52.6 % — BCSV re-serializes
   repeated strings every row (ZoH helps only for *unchanged* consecutive values).
3. **String-heavy / noisy-float writes**: 1.2–5× faster where dictionary lookup beats per-row
   string copies (string_heavy, high_cardinality_string, sensor_noisy on volatile data).
4. **Noisy-float compression**: byte-stream-split-style wins — sensor_noisy 15.7 % vs 34.2 % (ts).
5. **Ecosystem & parallelism**: multi-threaded readers, predicate pushdown with tuned row groups,
   Arrow/DuckDB/Spark interop, nulls as first-class.

## 4. Proposal — logical next steps

Ordered so that correctness lands first, nothing sacrifices the write-path speed advantage, and
each step is independently shippable. Guard every step with the interleaved benchmark
(`run.py interleaved`) to prove no regression.

### Phase 1 — Critical fixes (no format change, days)
1. Batch codec synchronization: H1 (stream ownership + EOF via flag), H2 (exception slot under
   mutex), M6 (unconditional rethrow in `finalize()`). Add a ThreadSanitizer CI job for the batch
   codec tests.
2. Write-side `MAX_ROW_LENGTH` check in `writeRow()` (H3).
3. Delta002 decode code validation (M1); flat001 span trim (M3); footer `start_offset` range check
   and batch-size cross-check vs header packet size (M4); packet checksum validation in
   direct-access `loadPacket` (M2).
4. `static_assert` little-endian (M5) + fix stale comments; make expected static-layout skips not
   poison the macro exit code.

### Phase 2 — Enable NaN/Inf properly (small, days)
5. Bit-compare change detection in static ZoH/Delta (fixes −0.0 corruption, restores NaN hold);
   `isfinite` guard in csv2bcsv; pandas `nan_policy` (preserve float NaN by default); NaN/Inf/−0.0
   /subnormal test matrix across codecs × layouts; document the bit-exactness guarantee.

### Phase 3 — Compression/encoding upgrades within the current packet format (weeks)
6. **Delta002 header suppression** (repeat-header → 0/1-byte row): removes the per-row header
   floor for idle channels; mirrors the existing ZoH length-0 path. Cheap, helps the 1000-ch idle
   case directly.
7. **Per-packet column min/max statistics** (optional footer section): packet skipping for
   time-range and predicate queries — this is what makes `read_pred` and `sample_every_n`
   *seek-class* instead of scan-class, at 8 MB granularity. Backward compatible.
8. **zstd batch file codec** (new FileFlags bit): the single cheapest ratio lever; delta output is
   exactly what an entropy stage eats. Closes most of the noisy-float and string gaps vs
   parquet+zstd while keeping LZ4 for embedded/streaming. BG-thread design absorbs the CPU cost.
9. **Per-packet string dictionary** (planned ToDo item 27): closes the 2–3× string-size gap and
   speeds string reads; enables Arrow dictionary export in pybcsv (zero-copy categoricals).

### Phase 4 — Hybrid row-columnar packets ("Packet002", the structural move, v2.0)
10. The batch codec already buffers whole 8 MB packets before compression: transpose rows into
    per-column chunklets (+ per-column encoding: delta/BSS for floats, dict for strings,
    RLE-bitpack for bools; per-chunklet offsets) on the background thread at flush. Streaming
    `writeRow()`, crash resilience, and packet index are untouched; embedded profiles keep the
    row-wise codecs. This is the only item that closes the 10–50× sparse-column gap, and it
    unlocks byte-stream-split + SIMD type-homogeneous decode loops (ToDo item 28) as by-products.

### Non-goals for now
- Runtime codegen (ToDo 28's LLVM idea) — revisit only after Packet002.
- Multithreaded reads — worth designing after Packet002 (column chunklets parallelize naturally).

Priority rationale: Phases 1–2 close every correctness hole found without touching the wire
format. Phase 3 items 7–9 each buy a measured parquet gap back cheaply. Phase 4 is the one
architectural investment that changes the competitive picture (sparse columns) rather than
narrowing it.
